## Context

`docs/03-data-model.md` is the agreed target shape and validation rule set for Season, Player, Team, Game, and Publishes. `docs/02-architecture.md` already commits to Zod as the validation library, shared between client and server, so forms and server actions/route handlers validate identically. Nothing in the codebase implements this yet ‚Äî `seed/types.ts` is a deliberate, explicitly temporary stand-in used only by the seed module.

This design was scoped through an explore-mode discussion that resolved several tensions between the data-model doc's validation rules and what a single-document Zod schema can actually enforce.

## Goals / Non-Goals

**Goals:**
- One Zod schema per entity (Season, Player, Team, Game, Publishes) plus nested Game sub-shapes (Goal, Penalty, OpponentGoal, OpponentPenalty), importable by both server code and client forms.
- One Zod enum per documented enum (Position, GameType, GoalType, PenaltyCode), matching `docs/03-data-model.md` exactly.
- Enforce every validation rule from the data model doc that can be checked from a single document's contents alone.
- Field-level error messages precise enough to show directly in a form, identical on client and server because both call the same schema.

**Non-Goals:**
- Uniqueness/cross-document checks (`number` unique among active players; blocked roster removal when a player is referenced elsewhere). These require a DB query and belong to KAN-14 or the service layer.
- Restricting which `GameType` values are offered at game-creation time (`NIHC` historical-only). That's a UI concern for the game-creation form (KAN-18), not a schema concern.
- ID generation, audit-timestamp stamping, repositories ‚Äî all KAN-14.
- Migrating `seed/types.ts` onto these schemas.

## Decisions

**Zod v4.** Faster than v3, and its reworked error-customization API (`error` param replacing `message`/`errorMap` fragmentation) is a better fit for "clear field-level messages" than v3's. No existing code depends on v3 behavior, so there's no migration cost to avoid.

**Module location: `lib/schemas/`.** Sits next to the existing `lib/mongodb.ts` and `lib/s3.ts`, keeping all "shared, no I/O" modules under `lib/`. One file per entity (`season.ts`, `player.ts`, `team.ts`, `game.ts`, `publish.ts`), one file for enums (`enums.ts`), re-exported from `lib/schemas/index.ts`.

**Single flat `GameType` enum, no create/stored split.** `docs/03-data-model.md` says `NIHC` is "valid only on historical records," which is a temporal/workflow constraint, not a shape constraint ‚Äî the field always holds one of 4 string values regardless of when the document was created. Modeling that as two schemas (`GameTypeCreate` vs `GameTypeStored`) would ripple a create/stored distinction through every consumer of the type for a rule that's really "don't offer this option in this one form." Decision: one `GameType` enum with all 4 values, always valid at the schema level; the create-form allowlist is KAN-18's problem.

**Base/create schema split for entities, not enums.** IDs (`PLR######`, `GME######`, ‚Ä¶) and `createdAt`/`updatedAt` are generated by KAN-14's data-access layer, not known at the moment a client form is validated. Each entity gets a base shape schema (all fields, used for reads/stored documents) and a derived `*CreateInput` schema (`.omit({ _id, createdAt, updatedAt })`) for what a form/create action validates before those are assigned. This is a standard Zod pattern, not a new architectural concept, and doesn't conflict with "keep the typing simple" (that decision was specifically about not forking the `GameType` enum itself).

**Same-document rules enforced via `superRefine`.** Roster-membership checks (goal `scoredBy`/`assist1`/`assist2`, penalty `offender`, `netminderPlayerId`, `manOfTheMatchPlayerId`, `warriorOfTheGamePlayerId` must all be in `team.roster`) are cross-field but same-document, so they're implemented as a `superRefine` on the `Game` schema, attaching the error to the specific offending path (e.g. `team.goals[2].scoredBy`) rather than a generic document-level error.

**Cross-document rules documented, not implemented, with a code comment pointing at their real owner.** `number` uniqueness and blocked-removal are named directly in `docs/03-data-model.md`'s validation rules section, so a reader of this module could reasonably expect them here. Each schema/service boundary gets a one-line comment (not a docstring) noting the rule exists but is enforced by the unique partial index / KAN-14 service layer, so nobody re-derives or forgets it.

**PenaltyCode enum ships as documented, including the unconfirmed `HOST` label.** `docs/03-data-model.md` is being treated as the single source of truth for this ticket; resolving the *(confirm)* marker on `HOST` ("Holding the Stick") is a data question, not a schema question, and is left to whoever owns that doc.

## Risks / Trade-offs

- **[Risk]** A future contributor adds a uniqueness check directly into a Zod `.refine()` on `Player` (Zod refinements can be async, so it's technically possible to smuggle a DB call in) ‚Üí **Mitigation**: the code comment at the `number` field plus this design doc's Non-Goals should make the boundary explicit; the module has no DB import at all, which makes an async-DB refinement visibly out of place in review.
- **[Risk]** `GoalType`/`PenaltyCode` values drift from `docs/03-data-model.md` over time (e.g. a new penalty code added to the doc but not the enum) ‚Üí **Mitigation**: none automated in this ticket; the enum is a direct transcription and should be kept that way by convention. Worth a follow-up ticket if drift becomes a real problem.
- **[Trade-off]** Base/create schema split doubles the number of exported schemas per entity ‚Üí accepted, since it's the standard way to express "this field doesn't exist yet" in Zod without making every field on the base schema optional (which would weaken validation for reads).

## Migration Plan

Additive only ‚Äî no existing code imports or depends on this module yet. No rollback concerns beyond reverting the commit.
